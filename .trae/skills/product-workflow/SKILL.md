# 产品工作方法论助手

这是一个系统化的产品工作方法论skill，帮助你从需求到原型全流程地思考和梳理产品。

## 核心工作流程

```
需求探索 → 业务梳理 → 原型设计 → 总结迭代
    ↓           ↓           ↓           ↓
  深度对话    流程图      低保真      方法沉淀
```

## 阶段一：需求探索（深度对话）

### 对话目标
通过系统化的提问，理解：
- 项目背景和驱动力
- 目标用户和场景
- 核心问题和机会点
- 成功标准和约束

### 关键问题框架

#### 1. 背景理解
- 这个项目是为了解决什么问题？
- 这个问题为什么重要？
- 当前是如何解决这个问题的？
- 有什么痛点？

#### 2. 用户洞察
- 目标用户是谁？
- 用户的典型使用场景是什么？
- 用户当前的工作流程是怎样的？
- 用户的决策链条是怎样的？

#### 3. 业务目标
- 期望达到什么效果？
- 如何衡量成功？
- 项目的时间范围和里程碑？
- 资源和预算约束？

#### 4. 边界和约束
- 什么是必须有的？
- 什么是一定不能有的？
- 技术或业务的限制？

### 对话原则
- 每个问题后，根据用户回答追问"为什么"和"例如"
- 记录关键洞察和假设
- 在适当时机总结理解，验证一致性
- 发现矛盾时，引导用户澄清

## 阶段二：业务流程梳理

### 流程图类型选择

#### 1. 泳道图（推荐）
适用于：涉及多角色/多系统的业务流程
能够清晰展示：各角色的职责和交互

#### 2. 功能流程图
适用于：单一功能或简单流程
能够清晰展示：操作步骤和判断逻辑

#### 3. 用户旅程图
适用于：需要强调用户感受和场景
能够清晰展示：用户在各阶段的情绪和接触点

### 流程图要素

| 要素 | 说明 | 示例 |
|------|------|------|
| 角色/系统 | 流程中的参与者 | 用户、管理员、系统 |
| 动作/操作 | 具体的行为 | 提交、审批、通知 |
| 判断节点 | 分支条件 | 审核是否通过 |
| 数据/信息 | 流转的内容 | 订单、审批意见 |
| 流程走向 | 顺序和分支 | →、↘、↙ |

### 梳理步骤
1. 识别角色：列出所有参与者
2. 识别场景：从用户视角描述完整场景
3. 拆解步骤：每个角色的每个动作
4. 标注判断：条件分支和异常流程
5. 确认边界：系统边界和集成点

## 阶段三：原型设计辅助

### 设计原则
- 先穷举：列出所有可能的功能点
- 再收敛：基于优先级和资源做减法
- 保真度：从低到高，快速迭代

### 原型设计流程

#### 1. 信息架构
- 梳理功能模块
- 确定导航结构
- 明确层级关系

#### 2. 页面结构
- 每个页面的核心目标
- 页面内的信息优先级
- 页面间的关系和跳转

#### 3. 交互设计
- 主要操作流程
- 反馈机制
- 异常和边界处理

### 输出物
- 页面清单
- 核心流程的线框图
- 关键交互说明

## 阶段四：总结迭代

### 每个阶段的产出物

| 阶段 | 产出物 |
|------|--------|
| 需求探索 | 需求背景文档、用户画像、问题陈述 |
| 业务梳理 | 流程图、泳道图、用户旅程图 |
| 原型设计 | 信息架构、线框图、交互说明 |

### 迭代机制
- 每完成一个阶段，停下来回顾：
  - 新发现了什么？
  - 之前假设是否被验证或修正？
  - 有什么需要调整的？
- 记录未解决的问题和风险

### 方法论沉淀
- 总结本次项目的成功经验和教训
- 提炼可复用的方法和模板
- 更新最佳实践库
